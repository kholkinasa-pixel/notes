// cheatsheet

Git — памятка
для новичка

Только нужные команды. Коротко, по делу, с примерами.

01 — Настройка
git config
Представиться гиту
Твоё имя и почта будут видны в истории коммитов. Делается один раз при установке.
git config --global user.name "Имя"
git config --global user.email "email@mail.ru"
02 — Создание репозитория
git init
Создать репозиторий с нуля
Превращает обычную папку в git-репозиторий. Запускается один раз в начале проекта.
git init
git clone
Скачать чужой репозиторий
Копирует проект с GitHub (или другого сервера) к тебе на компьютер со всей историей.
git clone https://github.com/user/repo.git
03 — Сохранение изменений
git status
Что сейчас происходит?
Показывает какие файлы изменены, какие готовы к коммиту, а какие ещё нет. Используй постоянно.
git status
git add
Подготовить файлы к коммиту
Говоришь гиту: «эти файлы я хочу сохранить». Без этого коммит не получится.
git add index.html  # один файл
git add .          # все изменения
git commit
Сделать «снимок» проекта
Сохраняет текущее состояние файлов в историю. Всегда пиши понятное сообщение — что изменил и зачем.
git commit -m "Добавил форму входа"
git diff
Что именно изменилось?
Показывает построчно, что ты добавил или удалил с последнего коммита. Удобно перед коммитом.
git diff
04 — История и отмена
git log
Посмотреть историю коммитов
Показывает все сохранения: кто, когда и с каким сообщением. Флаг --oneline делает список компактным.
git log --oneline
git restore
Отменить изменения в файле
Возвращает файл к состоянию последнего коммита. Пригодится если что-то сломал и хочешь откатить.
git restore index.html
git reset
Убрать файл из подготовленных
Отменяет git add — файл остаётся изменённым, но уже не попадёт в следующий коммит.
git reset HEAD index.html
05 — Ветки
git branch
Список веток / создать ветку
Ветки — это параллельные версии проекта. Делаешь новую фичу — создай ветку, не ломай основную.
git branch             # список
git branch feature-login  # создать
git switch
Переключиться на другую ветку
Переходишь работать в другую ветку. С флагом -c создаёт и сразу переключается.
git switch main
git switch -c new-feature  # создать + перейти
git checkout
Старый способ переключения — умеет много
Раньше делал всё: переключал ветки, создавал их, восстанавливал файлы. Сейчас его заменили switch и restore, но ты будешь встречать checkout в туториалах — знай, что это одно и то же.
git checkout main               # перейти на ветку
git checkout -b new-feature     # создать + перейти
git checkout index.html         # откатить файл
git merge
Слить ветку в текущую
Объединяет изменения из другой ветки в ту, где ты сейчас находишься. Если один и тот же файл правили в обеих ветках — возникнет конфликт, его нужно разрешить вручную.
git switch main
git merge feature-login  # влить feature-login в main
06 — Отмена коммитов
git revert
Отменить коммит безопасно
Создаёт новый коммит, который отменяет старый — история не теряется. Это безопасный способ откатить изменения, особенно если уже сделал push на GitHub. В отличие от reset, revert не переписывает историю.
git log --oneline               # смотришь хэш коммита
git revert a3f5c12              # отменяешь его
07 — Работа с GitHub
git push
Отправить коммиты на GitHub
Загружает твои сохранения на удалённый сервер. Делай это регулярно — это твой бэкап.
git push origin main
git pull
Скачать изменения с GitHub
Подтягивает чужие (или твои с другого компьютера) коммиты. Делай перед началом работы.
git pull origin main
08 — Pull Request
Что такое PR?
Запрос на слияние твоей ветки
Pull Request (PR) — это предложение добавить твои изменения в основную ветку. Ты как бы говоришь команде: «смотрите, что я сделал — проверьте и примите». PR живёт на GitHub, а не в гите — это интерфейс для code review.
# Всё делается через сайт GitHub, не через терминал
Шаг 1
Создай ветку и запушь её
Никогда не работай напрямую в main. Создай отдельную ветку для своей задачи, сделай коммиты и отправь ветку на GitHub.
git switch -c fix-header-bug
# ... делаешь изменения ...
git add .
git commit -m "Исправил баг в шапке"
git push origin fix-header-bug
Шаг 2
Открой PR на GitHub
Зайди на страницу репозитория — GitHub сам предложит кнопку «Compare & pull request». Нажми, напиши название и описание: что сделал и зачем. Выбери, в какую ветку хочешь влить (обычно main).
# GitHub покажет кнопку сам после push'а
# Или: вкладка Pull Requests → New pull request
Шаг 3
Code review — коллеги смотрят твой код
Другие разработчики оставляют комментарии прямо к строкам кода. Ты отвечаешь, вносишь правки и делаешь новые коммиты в ту же ветку — PR обновится автоматически.
# Внёс правки после ревью:
git add .
git commit -m "Учёл комментарии ревью"
git push origin fix-header-bug
Шаг 4
Merge — PR принят и влит
Когда все довольны, кто-то нажимает «Merge pull request» на GitHub. Твои изменения попадают в main. После этого ветку можно удалить — она больше не нужна.
# После merge — удали ветку локально:
git switch main
git pull origin main
git branch -d fix-header-bug
09 — Типичный рабочий день
Стандартный цикл работы выглядит так:
git pull → поработал → git add . → git commit -m "..." → git push → открыл PR
